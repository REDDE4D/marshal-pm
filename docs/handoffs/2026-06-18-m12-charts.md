## What was built

### Task 1 ŌĆö `internal/server/stores.go`, `server.go`

Extracted the metric-history query logic out of `*Server.FleetMetricsHistory` into a method
on `*stores`:

```go
func (s *stores) History(agent, selector string, sinceMs, bucketMs int64) ([]metricstore.Bucket, error)
```

Matches the selector exactly or as a `selector#` prefix, merges across instances
(`metricstore.MergeBuckets`), oldest first; `sinceMs` is a window width
(`lowerMs = now - sinceMs`); `bucketMs` passes through `metricstore.AutoBucketMs`; a missing
agent returns `(nil, nil)`. `FleetMetricsHistory` is now a thin proto wrapper that keeps its
`NotFound` guard and default-since behavior. Behavior-preserving ŌĆö the existing
`TestFleetMetricsHistory` is the regression guard.

### Task 2 ŌĆö `internal/dashboard/metrics.go` (new) + wiring

New session-guarded route `GET /api/metrics`:
- `?since=<ms>` (default 300000 = 5 m) ŌåÆ **batched**: recent buckets for every agent/proc in
  the live fleet (sparklines).
- `?agent=X&selector=Y[&since=<ms>][&bucket=<ms>]` (default since 3600000 = 1 h) ŌåÆ a single
  proc's series (detail panel).

JSON: `[{agent, procs:[{name, buckets:[{ts, cpu_avg, cpu_max, mem_avg, mem_max}]}]}]`.

The dashboard defines a `MetricsHistory` interface (satisfied structurally by `*server.stores`
ŌĆö no import cycle; dashboard imports only the leaf `metricstore` package). `newHandler` /
`NewHandler` / `Serve` gained a `MetricsHistory` parameter; `ServeDir` passes the in-scope
`ss` into `dashboard.Serve(ctx, httpAddr, reg, ss, auth, cert)`. `parseMs` defaults empty/
invalid/negative query values.

### Task 3 ŌĆö `web/src/Sparkline.tsx` (new), `api.ts`, `Fleet.tsx`

`Sparkline` is a pure presentational SVG `<polyline>` (autoscale minŌåÆmax, flat-line fallback,
empty ŌåÆ blank svg). `api.ts` gained `getMetrics(sinceMs)` and
`getMetricsForProc(agent, selector, sinceMs, bucketMs)` (both throw on 401, like `getFleet`).
`Fleet.tsx` gained CPU and Mem columns (numeric value + sparkline) and a **separate** 10 s
metrics poll (the existing 2 s fleet poll is untouched). The metrics poll is best-effort ŌĆö
a failure does not log the user out (only the fleet poll owns auth).

### Task 4 ŌĆö `web/src/MetricChart.tsx` (new), `Fleet.tsx`, `styles.css`

`MetricChart({buckets, metric})` draws an SVG with min/max Y gridlines + labels, a faint max
series and a solid avg series; empty buckets ŌåÆ "No history yet." Each process row is
clickable (`<Fragment>`-wrapped row + panel) and toggles a single open detail panel showing
5m/1h/6h/24h window buttons (`e.stopPropagation()` so they don't toggle the row) and two
charts. A detail-poll `useEffect` runs only while a panel is open, with `[expanded, windowMs]`
deps (changing either refetches immediately), cleaned up on close/unmount, best-effort.

The built SPA lives in the committed `internal/dashboard/dist/`; `go build` embeds it without
a Node toolchain. Rebuild with `make ui` after any `web/src/` change.

## Live-demo result (2026-06-18, scratch `XDG_DATA_HOME=/tmp/marshal-m12-demo`, server `:9200`/`:9201`)

End-to-end verified against a real server with two enrolled demo processes (`busy`, a CPU
loop; `ticker`, idle):

1. `GET /api/metrics` without a cookie ŌåÆ **401**; after login ŌåÆ **200**. SPA shell + assets serve.
2. Batched `GET /api/metrics?since=300000` ŌåÆ buckets for both procs at 5 s spacing:
   `busy` CPU Ōēł **71ŌĆō75 %** (varying ŌĆö a meaningful sparkline), `ticker` CPU Ōēł **0.03 %**
   (near-flat), both with memory (~2ŌĆō3 MB).
3. Single-series `?agent=demo-1&selector=busy&since=300000&bucket=0` ŌåÆ 10 buckets of real CPU;
   `since=3600000` (1 h) ŌåÆ 2 wider buckets (correct `AutoBucketMs` re-bucketing).
4. Unknown agent `?agent=ghost&...` ŌåÆ `[{"agent":"ghost","procs":[{"name":"busy","buckets":[]}]}]`,
   HTTP **200** (graceful empty ŌåÆ "No history yet" in the panel).
5. Teardown: scratch processes deleted, scratch daemon stopped, scratch dir removed; no
   `/tmp/marshal-m12-demo` processes remain.

**Note (not an M12 issue):** a separate, pre-existing user daemon
(`/Users/sebastiankuprat/process manager/marshal daemon`, PID seen at demo time) was running
in the **default** state dir managing a `clock` app since before this session. It was left
untouched ŌĆö it is the user's own process, not a demo orphan.

The browser chart rendering itself was not screenshotted in this run; it was verified
indirectly by confirming the exact data the React components consume (the `/api/metrics`
payloads above) and that the embedded SPA shell + hashed assets serve. A visual browser pass
is a reasonable optional follow-up.

## Deferred / known issues

Out of scope for M12 (per the spec), carried forward:

1. **X-axis time-tick labels** on `MetricChart` ŌĆö the spec prose mentioned "a few X-axis time
   tick labels" as a nice-to-have; the chart currently has Y gridlines + labels but no X
   ticks. Minor under-delivery against the prose, not the milestone scope.
2. **Hover tooltips** with exact point values; **chart zoom/pan**.
3. **Persisted UI state** ŌĆö the expanded panel and selected window reset on reload.
4. **Sparkline scaling is relative** (each series autoscales to its own minŌåÆmax), so a flat-
   nonzero and a flat-zero series look alike and CPU/mem sparklines aren't absolute-comparable.
   Intentional per spec ("autoscale minŌåÆmax").
5. **Single-series test coverage** ŌĆö `TestMetricsSingleSeries` asserts routing (agent/selector)
   but not that `since`/`bucket` are forwarded to `History` (the `fakeMetrics` only records
   agent/selector). Low risk; a follow-up test would close it.

Carried over from M11 and still open: in-memory sessions lost on restart; `server passwd` /
`token --rotate` not hot-reloaded; no login rate-limiting; self-signed cert warning;
read-only dashboard (no process controls); no live log tailing; single-user accounts.
