- Everything builds CGO-free; `go test ./... -race` is green; gofmt/vet clean; 39 web tests pass.

## What shipped this milestone

### Distribution / install methods
- **`go install`** works: the Go module path was renamed `marshal` → `github.com/REDDE4D/marshal-pm`
  (mechanical rewrite of go.mod, 98 import sites, and the version-stamp ldflags path in the
  Makefile + .goreleaser.yaml).
- **`.deb` / `.rpm`** via GoReleaser `nfpms` — each carries the binary (`/usr/bin/marshal`), shell
  completions (bash/zsh/fish, generated by a `before` hook), and a **disabled-by-default**
  `marshal.service` (`packaging/marshal.service`, runs `marshal daemon` under `/var/lib/marshal`).
- **Docker** — multi-arch (amd64+arm64) image of the **central fleet server + dashboard** at
  `ghcr.io/redde4d/marshal` (`packaging/Dockerfile`, alpine + ca-certificates + openssh-keygen).
  NOT the per-host agent. The release workflow gained QEMU + buildx + a GHCR login and
  `packages: write`.

### Update notifier (server + dashboard)
- **`internal/updatecheck`** — `Outdated(current, latest)` (numeric semver; never nags
  `0.0.0-dev`/local/git-describe builds) and `fetchLatest` (one anonymous GET to GitHub's
  `/releases/latest` **302 redirect** — deliberately NOT the rate-limited API; no identifiers
  sent). A `Checker` caches a daily snapshot; `Run(ctx)` polls every `DefaultInterval` (24h);
  opt-out via `MARSHAL_NO_UPDATE_CHECK`.
- **Server**: creates + runs the checker (dashboard mode only) and exposes
  `GET /api/update` → `{enabled,current,latest,outdated,checked_at,outdated_agents}`. Outdated
  agents are computed by comparing each agent's `marshal_version` (already in the registry) to
  the latest release.
- **Dashboard**: `web/src/UpdateBanner.tsx` — a dismissible hairline banner in the app shell,
  dismissal remembered per-latest-version in localStorage, styled in the instrument language.
  **No self-update** — the binary is never replaced; users update via their install method.

## Key decisions / non-obvious bits

- **Formula, not cask** for Homebrew (`brews` in .goreleaser.yaml): casks are macOS-only; a
  formula lets `brew install` work on Linux too. GoReleaser deprecated `brews`, so CI pins
  `goreleaser ~> v2`. (Same story now applies to `dockers`/`docker_manifests` → `dockers_v2`.)
- **Redirect, not API** for the update check — we hit GitHub's API 403 rate-limit during
  development; the `/releases/latest` redirect isn't rate-limited and needs no token.
- **Consciously deferred** (from the earlier security review): in-process ed25519 keygen (would
  add an `x/crypto` dep for a Low item) and an **active gRPC auth lockout** (per-IP keying has a
  NAT-shared-fleet hazard — needs a design pass, e.g. high threshold or global rate-limit).

## Build / run / test

```bash
make build                         # stamps version from git tags
go test ./... -race -count=1       # all green
cd web && npm run build            # rebuild the embedded dashboard (writes internal/dashboard/dist)
goreleaser release --snapshot --clean   # local dry-run of the release pipeline (skip docker if no daemon)
```

Release flow (used for v0.5.0): promote `[Unreleased]`→`[X.Y.Z]` in CHANGELOG + compare links on
`dev`, commit, merge `dev`→`main` (`--no-ff`), `git tag vX.Y.Z`, push `main` + `dev` + tag. The
`Release` workflow then runs GoReleaser (binaries + checksums + deb/rpm + Docker + brew formula).

## Deferred / known issues

- Active gRPC auth lockout + in-process ed25519 keygen (see above).
- The update notifier is dashboard/server-only; no CLI hint was added (intentionally — chosen
  scope was "server + dashboard").
- `dockers`/`docker_manifests` and `brews` are GoReleaser-deprecated; revisit at the v3 boundary
  (cask split for mac vs. keeping a Linux-capable formula; `dockers_v2`).
- Vite emits a >500 kB chunk-size warning for the dashboard bundle (pre-existing; cosmetic).
