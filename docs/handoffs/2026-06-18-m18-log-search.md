## What was built

### 1. Core filter — `internal/logstore` (commit a88c5aa)

- `escapeLike(s)` backslash-escapes `\ % _` so the needle is a pure literal.
- `Tail` and `Since` gained a `text string` argument. When non-empty the SQL gains
  `AND text LIKE ? ESCAPE '\'` with parameter `"%" + escapeLike(text) + "%"`. SQLite `LIKE`
  is case-insensitive for ASCII — matching the prior client-side box. **Empty `text` ⇒ no
  clause (unchanged behavior).** The clause is appended in lock-step with its arg, so
  placeholder order stays correct in both of `Since`'s branches (follow `rowid > ?` and
  backfill `ORDER BY rowid DESC`).
- The new arg was threaded through every caller (`logStores.Since`, the `LogsHistory`
  interface, all production + test call sites) passing `""` where not yet wired — no behavior
  change in that commit.

### 2. Dashboard surface (commits 92d222c, 153095a)

- `/api/logs` reads a new `q` param and passes it to `Since` (`internal/dashboard/logs.go`).
- Frontend (`web/src`): `getLogs` sends `q`; `Fleet.tsx` debounces the search box (250 ms) and
  restarts the log poll with `after=0` on a query change (fresh backfill of matching lines from
  full history, then the live tail is filtered too); `LogView.tsx`'s client-side
  `lines.filter(...)` was removed — the server is now authoritative.

### 3. CLI / gRPC surface (commits 2367a76, 5ab7e60)

- Added `string grep = 5;` to `FleetLogsHistoryRequest` (`proto/marshal/v1/fleet.proto`);
  regenerated `internal/pb` via `go generate ./internal/pb/` (protoc + plugins are installed).
- `FleetLogsHistory` passes `req.GetGrep()` into `store.Tail`.
- `fleet logs --grep <text>` sets the field.

### Deferred / known issues (Minor — accepted)

- Frontend: `q=` is always sent (empty = no filter, inert); `logSearchDebounced` inits to `""`
  rather than mirroring `logSearch` (cosmetic — both start `""`).
- Substring `LIKE` is a table scan within the label-narrowed row set. Fine for self-hosted
  scale; an FTS5 index is the documented future optimization if a store grows very large.

### Out of scope (per the spec)

- Regex / FTS5 / fuzzy matching; match highlighting in the UI; time-range filters; multi-agent
  (cross-agent) search; persisting search history. Search stays scoped per agent+selector.

## Live-demo result (2026-06-18, scratch `/tmp/marshal-m18-demo`, server `:19380`/`:19381`)

Seeded an agent store (`agents/dev-1/logs.db`, 7 lines: 2 `ERROR`, 1 with `timeout`, 1 with a
literal `95%`) via a throwaway `cmd/marshalseed` helper (since `logStores.has` only checks the
dir exists, no full agent was needed). Then a real server with `--http-listen`:

- **CLI (real gRPC):** `fleet logs dev-1 web --grep error` → the 2 `ERROR` lines
  (case-insensitive — `error` matched `ERROR`); `--grep timeout` → 1; `--grep '95%'` → the
  literal WARN line (the `%` did **not** act as a wildcard); no `--grep` → all 7.
- **Dashboard (`/api/logs?q=`, the endpoint the search box calls):** no `q` → 7 lines;
  `q=error` → 2; `q=timeout` → 1; `q=95%` (URL-encoded) → 1 literal. Raw JSON confirmed the two
  matching lines.
- Teardown: server stopped, scratch + seeder removed; `pgrep -fl marshal` shows only the user's
  pre-existing daemon (pid 84457). No demo orphans.

**Note on the rendered UI:** the search *behavior* was proven end-to-end through the dashboard's
real HTTP endpoint (which the React search box calls) and `make ui` builds the box cleanly, but
the box itself was not screenshotted in a live browser this session (self-signed-TLS + login
friction). A quick in-browser view can be spun up on request.
