## What exists now (and works)

A daemon + control CLI. Quick proof it works (uses a short `/tmp` base because macOS caps
Unix socket paths at ~104 bytes — see Gotchas):
```bash
go build -o marshal ./cmd/marshal
export XDG_DATA_HOME=/tmp/msmoke && rm -rf "$XDG_DATA_HOME"
printf 'apps:\n  - name: clock\n    cmd: sh\n    args: ["-c","while true; do date; sleep 1; done"]\n    instances: 2\n' > /tmp/m.yaml
./marshal start /tmp/m.yaml   # auto-spawns the daemon, prints 2 procs
./marshal list                # clock#0, clock#1 online with pid/uptime
./marshal restart clock
./marshal save                # writes $XDG_DATA_HOME/marshal/dump.json
./marshal delete all
./marshal resurrect           # clock comes back from dump.json
./marshal kill                # daemon stops all procs and exits; socket removed
```

### Architecture
```
  marshal (CLI)  ──gRPC over UDS──▶  marshald (daemon)
   thin client                        owns manager.Manager + all processes
                                      $XDG_DATA_HOME/marshal/marshald.sock
                                      (default ~/.marshal/marshald.sock)
```

### Packages (new in M2)
- `proto/marshal/v1/daemon.proto` + `internal/pb` — the gRPC `Daemon` contract and
  generated code (committed). Service: Start/Stop/Restart/Delete/List/Describe/Save/
  Resurrect/Kill, plus a **defined-but-unimplemented** `Logs` server-stream and
  dormant `ProcInfo.cpu`/`mem` (reserved for M3).
- `internal/store` — state-dir layout (`$XDG_DATA_HOME/marshal` or `~/.marshal`) and
  atomic `dump.json` (save/resurrect). Dir is created `0o700`.
- `internal/daemon` — the gRPC server (`Server` impl + `Run(ctx, st)` serve loop): binds
  the socket (chmod `0o600`), auto-resurrects from `dump.json` on startup, maps RPCs →
  manager ops with proper gRPC status codes, graceful shutdown on ctx-cancel or `Kill`.
- `internal/client` — `Connect(st)`: dials the socket; if nothing is listening, auto-spawns
  `marshal daemon` detached (`Setsid`, output → `marshald.log`), waits for readiness.
- `cmd/marshal` — cobra commands: `daemon` (hidden, runs the server), `start`, `stop`,
  `restart`, `delete`, `list`/`ls`, `describe`, `save`, `resurrect`, `kill`, and the
  preserved M1 `run`.

### Changed M1 packages
- `internal/manager` — **refactored from static to dynamic.** Old `New(cfg)`+`Run(ctx)`
  is gone. New API: `New(ctx)`, `Add(config.App)`, `Stop/Restart/Delete/Describe(sel)`,
  `List()`, `Specs()`, `StopAll()`. Each instance has its own cancel + done channel.
  Mutations are serialized with an `opMu` (safe under concurrent gRPC dispatch); the
  short-lived `mu` protects state and keeps reads responsive during a blocking stop.
  Since `supervisor.Instance.Run` is one-shot, **Restart recreates instances** from the
  stored spec (so the restart counter resets to 0 on an explicit `restart`).
- `internal/config` — added `Duration` JSON (un)marshal, `json:` tags on `App` (so
  `dump.json` uses the same snake_case schema as `marshal.yaml`), and exported
  `Config.Prepare()` (defaults+validate), reused by the daemon to admit `AppSpec`s.
- `cmd/marshal/main.go` — `run` rebuilt on the dynamic manager; M1 `run` e2e unchanged.

## Key decisions / non-obvious things

- **gRPC over a Unix socket** (chosen deliberately over JSON/net-rpc) so the message
  contract is reused verbatim for the future sub-project-#3 agent↔server link.
- **Lifecycle-only milestone.** Logs and metrics were intentionally cut from M2 and live
  entirely in M3; the proto reserves them so M3 adds them without a contract break.
- **Selector semantics:** `name` | numeric `id` | `all`. `start` of an existing name is an
  error (use `restart`). Ids are monotonic from 1 (PM2-style).
- **`Describe` returns a `ProcList`** (an app has N instances), a refinement over the
  spec's "→ ProcInfo".
- **`Kill` uses `go s.kill()`** on purpose: `GracefulStop` waits for in-flight RPCs, so
  calling it inline from the Kill handler would deadlock on the Kill RPC itself.
- **Security model = filesystem perms** (per spec): state dir `0o700`, socket `0o600`.

### Regenerating the proto (only if you edit daemon.proto)
Generated code is committed, so normal builds need no toolchain. To regenerate you need
`protoc` plus the Go plugins on PATH (NOT pinned in a tools.go — a known gap):
```bash
brew install protobuf
go install google.golang.org/protobuf/cmd/protoc-gen-go@latest
go install google.golang.org/grpc/cmd/protoc-gen-go-grpc@latest
PATH="$(go env GOPATH)/bin:$PATH" go generate ./internal/pb
```

## Gotchas / environment notes

- **macOS Unix socket path limit (~104 bytes).** Tests that bind a socket must use a SHORT
  base dir (`os.MkdirTemp("/tmp", ...)`), NOT `t.TempDir()` (whose `/var/folders/...` paths
  overflow `sun_path`). The e2e tests already do this; remember it for any new socket test.
- The state dir nests everything under `$XDG_DATA_HOME/marshal/` (so with
  `XDG_DATA_HOME=/tmp/x`, the socket is `/tmp/x/marshal/marshald.sock`).
- Go via Homebrew at `/opt/homebrew/bin/go` (1.26.4). gRPC `v1.81.1`, protobuf `v1.36.11`.
- No git remote — local-only; PRs unavailable until one is added.

## Deferred / known issues (address in M3 or later)

- **CLI error prefix is verbose.** gRPC errors surface as
  `error: rpc error: code = NotFound desc = no app matching "x"`. Strip via
  `status.FromError` in `cmd/marshal/control.go` `withClient` so the user sees just the
  message. (Cheap M3 polish.)
- **Auto-spawn has no cleanup on timeout.** If `client.Connect`'s `waitReady` times out, the
  spawned daemon process is neither killed nor reaped. Consider a PID/lock file in M3.
- **`Resurrect` swallows per-app `Add` errors** (intended to skip already-running apps, but
  also hides genuine failures). An aggregated warning would be nicer.
- **`config` zero-value-as-unset:** an explicit `kill_timeout: 0s` is promoted to the 5s
  default by `applyDefaults`. Pre-existing semantic; now reachable via dump.json round-trips.
- **`UPTIME` shows `-` for non-online states** (only computed when online) — cosmetic.
- **Untrusted `dump.json`:** a tampered dump can make the daemon launch any `cmd`. Same
  trust boundary as the socket perms; fine for single-user, note for the fleet direction.
- **No `tools.go`** pinning the protoc plugins (see regeneration note above).
- **Windows still deferred** (process groups, sockets, `Setsid` detach differ).
