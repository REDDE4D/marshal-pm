## TL;DR

Added **deploy-from-git** to the dashboard. A user picks "from git" in the add-app modal,
gives a repo URL + optional build command; the chosen agent **clones → builds → runs** it as
a normal supervised app. Because clone+build is slow, the deploy is **asynchronous**:
`POST /api/apps` returns *accepted* and the deploy state (`cloning → building → online | failed`)
rides the existing ~2s fleet heartbeat as synthetic `ProcInfo` entries. Git apps also get
**redeploy** (`POST /api/apps/redeploy` → fetch latest on the pinned ref → rebuild → restart in
place, swapping only on a successful build) and **dismiss** of a failed deploy (a `delete`
control op that clears the deploy state + removes the checkout dir). Auth is the agent host's
own git credentials — **no secrets stored in Marshal**.

A git-deployed app is just a `config.App` that also carries a `Source`. The new machinery is one
agent component (`internal/deploy`) that performs clone/build out of band, then hands the
resolved spec to the existing `doStart` chain. On agent restart a git app re-runs straight from
its existing checkout (no rebuild); its `Source` persists in `dump.json` so redeploy can
fetch+rebuild later.

## Key decisions / non-obvious

- **Dedicated `ControlOp_Deploy`/`Redeploy`** (not overloading `Start`) — deploy is async/fire-and-
  forget (`ok:true` = accepted), `Start` is sync and returns procs.
- **Synthetic snapshot entries**: a deploying app has no OS process, so `fleetSnapshot()` appends
  synthetic `ProcInfo` (state `cloning/building/failed`, `source=git`, `detail`) until real
  instances exist; on success `doStart` runs and the synthetic entry is dropped.
- **Redeploy = `git fetch origin <ref>` + `git reset --hard FETCH_HEAD`** (ref defaults to `HEAD`).
  This handles branch / tag / default-branch; the initial clone uses `git clone` + `git checkout
  <ref>` (handles SHAs too). Earlier `git reset --hard origin/<ref>` only worked for branches.
- **Build runs via `sh -c`** so compound commands (`npm ci && npm run build`) work; build/clone
  output is piped to the app's per-app log sink (`<name>#0`) so it shows in the existing log view.
- **Marshal-managed credentials is the NEXT milestone** (deferred) — it layers onto the same
  "agent runs git" seam. This milestone relies on host git credentials.

## Review findings fixed before merge

- **C1 (Critical, path traversal):** app `name` was unvalidated → `../..` could escape the deploy
  root in `git clone` and `os.RemoveAll`. Fixed centrally in `config.validate()` (name allowlist).
- **I1 (Important):** redeploy was broken for tag/SHA refs (`reset --hard origin/<ref>`). Fixed to
  fetch-by-ref + `reset --hard FETCH_HEAD`.
- **Live-demo bug:** the dashboard `/api/fleet` `procView` DTO (`internal/dashboard/fleet.go`)
  dropped `source`/`detail`, so the **redeploy button never rendered** and failed-deploy reasons
  were blank in the browser. Added the fields + test (commit `8fc4b07`). No task had touched that
  file — only the end-to-end demo caught it.
- Minors fixed: deployer nil-guards in command.go; `dispatchApp` audit label per action;
  dedicated unsupported-source-type test.

## Live demo result (2026-06-19, scratch `/tmp/marshal-m21-demo`, server `:9000`/`:9001`)

Real fleet: server + connected agent `dev-1` (seed app). Deployed a **local Go git repo** (real
`git clone` + `go build`):
- `POST /api/apps` git source → `{ok:true}`; fleet showed `gitapp[building/git]` → `gitapp[online/git]`.
- Redeploy v1→v2→v3 (incl. clicking the **redeploy button in the rendered dashboard**): logs showed
  `git fetch … FETCH_HEAD`, `HEAD is now at <sha>`, rebuild, restart, new version running.
- Failing build (`badapp`) → `failed/git` with `detail:"build failed: exit status 1"`; full compiler
  output visible in the dashboard **log view**; **dismiss** (delete) cleared the card AND removed the
  checkout dir.
- In-browser (Vite `:5173` → `:9001`): Signal login/overview render; **redeploy button shows only on
  the git app**, not the command app; add-app modal's **Command|from git** toggle reveals repo/ref/
  build/subdir and hides the command-only working-dir.
- **Teardown:** agent + server + Vite stopped, scratch dir + `.claude/launch.json` removed,
  `/tmp/marshal-m21` removed. `pgrep -fl marshal` shows only the user's own daemon (pid 1312); no orphans.

## Deferred / known issues

- **Marshal-managed credentials** — secret storage + encryption + injection at clone time. The next
  milestone; slots onto the same git seam. (Its own brainstorm → spec → plan.)
- **Dashboard file manager + in-browser editor** — a further-future idea the user flagged.
- **Redeploy when the checkout dir is gone** (manually removed / data dir moved): `git fetch` in a
  missing dir → `failed` card, app keeps running. Graceful but no re-clone fallback. Minor.
- A pinned-SHA redeploy is a no-op (immutable) and may fail if the server disallows fetch-by-sha;
  acceptable.
- Still open from earlier: per-app log-retention fields in the form; edit-an-app; one-click port
  open; light-mode; dashboard `/api/audit` view.
