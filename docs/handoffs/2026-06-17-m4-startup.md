## What exists now (and works)

`marshal startup` / `marshal unstartup` generate, install, and remove a boot service that runs
`marshal daemon` at startup; the booted daemon auto-resurrects `dump.json` (already wired since
M3 — `internal/daemon/server.go` `Run` loads the dump on start). The init system supervises
`marshald` (`Restart=on-failure` / `KeepAlive`); `marshald` supervises the apps.

### Smoke proof (macOS host, this session)
```bash
go build -o marshal ./cmd/marshal
export XDG_DATA_HOME=/tmp/m4smoke

# --system: stages + prints, installs NOTHING
./marshal startup --system
#   -> Staged unit at /tmp/m4smoke/marshal/com.marshal.daemon.plist
#   -> prints `sudo cp ... && sudo launchctl bootstrap system ...`

# user-level: real install + uninstall round-trip (verified, left nothing behind)
./marshal startup            # writes ~/Library/LaunchAgents/com.marshal.daemon.plist + launchctl bootstrap gui/<uid>
launchctl list | grep com.marshal.daemon   # -> listed, running
./marshal unstartup          # launchctl bootout + deletes the plist
launchctl list | grep com.marshal.daemon   # -> gone
```

### Architecture (M4 additions)
```
  cmd/marshal/startup.go  (startup / unstartup commands)
     │  resolveConfig: os.Executable (abs) + user.Current + $HOME + $XDG_DATA_HOME + store.Dir + getuid
     ▼
  internal/startup  (LEAF package — imports only stdlib)
     ├─ Detect(GOOS) -> Platform        (darwin=launchd; linux=systemd iff /run/systemd/system; else error)
     ├─ Platform.InstallPlan / RemovePlan -> Plan{UnitPath, StagePath, Content, PostInstall[], PostRemove[], NeedsRoot}
     │     ├─ systemd.go  : renders the .service unit (user vs --system)
     │     └─ launchd.go  : renders the .plist (LaunchAgent vs LaunchDaemon), XML-escaped
     └─ side effects (driven by an injectable Runner):
           Apply        (user)   : write unit file 0644, then run PostInstall
           Remove       (user)   : run PostRemove (best effort), delete file (missing = ok, idempotent)
           StageAndPrint(--system): write to StagePath under the state dir, PRINT the sudo block (runs nothing)
```

### Four generated artifacts
- **systemd user** `~/.config/systemd/user/marshal.service` — no `User=`, `WantedBy=default.target`;
  PostInstall: `systemctl --user daemon-reload` / `enable --now` / `loginctl enable-linger <user>`.
- **systemd system** `/etc/systemd/system/marshal.service` — `User=<user>`, `WantedBy=multi-user.target`;
  staged + printed `sudo cp … && sudo systemctl daemon-reload && sudo systemctl enable --now …`.
- **launchd user / LaunchAgent** `~/Library/LaunchAgents/com.marshal.daemon.plist` — `RunAtLoad`+`KeepAlive`,
  no `UserName`; `launchctl bootstrap gui/<uid> <path>` (bootout to remove).
- **launchd system / LaunchDaemon** `/Library/LaunchDaemons/com.marshal.daemon.plist` — adds `UserName`;
  staged + printed `sudo cp … && sudo launchctl bootstrap system …`.

All four pin the absolute binary path and `HOME` (+ `XDG_DATA_HOME` only when set) so the booted
daemon resolves the same state dir.

## Key decisions / non-obvious things

- **`Cmd.String()` vs `Cmd.Display()`**: `String()` is the unquoted form used by the exec path
  (`ExecRunner` / test recorder, no shell involved). `Display()` shell-quotes for the human
  copy-paste output (the `--system` print + `unstartup --system`). Added during code review.
- **systemd `Environment=` quoting** quotes only when the value contains space/tab/`"`/`\`, and
  escapes `"`/`\` when it does — so normal paths render unquoted (stable output) while hostile
  values stay valid. launchd values are always XML-escaped via `encoding/xml`.
- **`--system` never self-escalates** — it stages the unit under the state dir and prints the
  exact `sudo` block. User-level installs directly (own home dir, no root).
- **`Detect(goos)` is parameterized** (GOOS + an injectable systemd probe) so all four branches
  are unit-tested without the host's real init system. Non-systemd Linux → clear error.
- **`RemovePlan` reuses `InstallPlan`**; only `UnitPath` + `PostRemove` are meaningful to `Remove`
  (commented in both platforms).
- Verified real-world: the binary path on this host contains a space
  (`/Users/sebastiankuprat/process manager/marshal`) and renders correctly because each
  `ProgramArguments` element is its own `<string>`.

## Deferred / known issues (address later)

- **No CLI-level test of the user-level install path.** The side-effecting logic (`Apply`/`Remove`
  writing the file + dispatching commands) IS unit-tested in `internal/startup/install_test.go`
  with a recording runner; the CLI glue hardcodes `startup.ExecRunner`, so a CLI test of the
  default path would need `resolveConfig` to stop sourcing `Home` from `user.Current().HomeDir`
  (to redirect it to a temp dir and avoid touching the real `~/Library/LaunchAgents`). Judged not
  worth a behavior change now. If revisited: add a `var newRunner = func(...) startup.Runner` seam
  + source `Home` from `$HOME`/`os.UserHomeDir`.
- **`unstartup --system` inlines its print loop** rather than reusing a `startup.PrintRemove`
  helper symmetric with `StageAndPrint`. Cosmetic.
- **Only systemd is supported on Linux** (OpenRC/SysV/runit error out by design). Windows still
  deferred (consistent with M1–M3).
- **launchd `bootstrap`/`bootout`** are the modern verbs; `launchctl load -w` is the documented
  fallback for very old macOS.
- Carried from earlier milestones (unchanged): the two deferred log items (max-line cap in
  `logs.Sink`; follow backfill→subscribe gap) belong to sub-project #2; M2/M3 items
  (auto-spawn cleanup on `waitReady` timeout; `Resurrect` swallows per-app `Add` errors;
  untrusted `dump.json` trust boundary).
