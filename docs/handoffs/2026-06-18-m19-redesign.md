## What was built

### Identity — "Signal"
Tokenized `web/src/styles.css` (CSS custom properties): bg `#0A0A0C`, panel `#121216`, border
`#26262C`, text `#C7CAD2`/dim/faint, accents cyan `#2DD4BF` + lime `#A3E635`, danger `#F87171`,
mem `#5B6BD8`. Monospace-forward: **JetBrains Mono vendored via `@fontsource/jetbrains-mono`**
(Vite bundles the woff2 into the embedded `dist` — offline, no CDN), `ui-monospace` fallback.
Prominent numbers (30px summary values), lowercase labels, flat. `Logo.tsx` = the bracket-`>`-
cursor mark + `marshal_` wordmark, reused on login + overview.

### Information architecture
- **Overview** (`Overview.tsx` + `SummaryCards.tsx` + `ProcessCard.tsx`): five fleet summary
  cards (agents online, processes up/total, total cpu %, total mem, **errors** = count of
  `state==="errored"`) + per-agent **full-width process cards** (status dot, name, state,
  meta line, cpu/mem sparklines, a **recent-error badge** when stderr-count>0, state-aware
  start/restart/stop, whole card links to detail; errored procs get a red left accent).
- **Process detail** (`ProcessDetail.tsx`): breadcrumb + back, header with controls, a stat-tile
  row (cpu · memory · uptime · **errors·5m**), a window selector (15m/1h/6h) over side-by-side
  cpu/mem charts, and the full log panel with stream filter + M18 server-side search.
- **Routing** (`router.ts`): a dependency-free hash router (`#/`, `#/a/<agent>/p/<proc>`) with
  back-button + deep-linking; unknown hash → overview.
- `ControlButtons.tsx` (shared): the **start button issues a `restart` op** (revives a
  stopped/errored process; no confirm — it's already down), restart/stop use the confirm step.
  Wrapper spans call both `stopPropagation` **and** `preventDefault` so a button click inside the
  card-`<a>` doesn't navigate.

### Backend (small)
- `logstore.Store.ErrorCounts(labels, sinceMs)` + `server.logStores.ErrorCounts(agent, sinceMs)`
  — per-label count of stderr lines with `ts >= sinceMs` (indexed, parameterized).
- `dashboard.LogsHistory` gained `ErrorCounts`; `GET /api/logstats?agent=<a>` (session-protected,
  best-effort, 5-min window) → `{"counts":{"web#0":3,…}}`. `NewHandler` signature unchanged.

### Task 1 dropped (design correction during implementation)
The plan/spec originally added a backend `start` control op. That was wrong: the proto's
`ControlOp_Start` carries a `StartRequest` (an app-spec list to launch *new* apps), not a by-name
selector. `manager.Restart` already revives a stopped/errored managed process, so the **start
button issues a `Restart`** — no backend control change, no dead code. The spec was corrected.

## Live in-browser demo (2026-06-18)

Stood up a real fleet stack (server `:9001` + a connected agent `dev-1`) running three apps:
`web` (healthy, emits stdout + periodic stderr `ERROR`), `worker` (quiet), and `importer`
(crashes → restarts repeatedly). Drove the SPA through the Vite dev proxy (`npm --prefix web run
dev` on `:5173`, proxying `/api` → `https://localhost:9001`) via the Preview tool. Observed:

- **Overview** rendered in Signal: `marshal_` header, five summary cards (prominent numbers,
  cpu in cyan), three full-width process cards with cpu/mem sparklines, **red recent-error
  badges** (`⚠ 25 errors` on web, `⚠ 66 errors` on the thrashing importer with its high restart
  count), and state-aware controls (start dimmed while online; restart/stop active).
- **Detail page** (clicked the web card): breadcrumb, header + controls, stat tiles incl
  `errors·5m`, the 15m/1h/6h window selector over side-by-side cpu/mem charts, and the log panel.
- **Server-side search**: typed `error` in the detail log search → the panel collapsed to only
  the red ERROR lines reaching back past the live tail (M18 full-history search, in the new skin).

Teardown: Vite + agent + server stopped, scratch dir / temp files / `.claude/launch.json`
removed; `pgrep -fl marshal` shows only the user's pre-existing daemon (84457); no demo orphans;
tree clean. (Note: the `importer` cycled starting→online→crash→restart and was usually caught in
`online`/`restarting` rather than the terminal `errored` state, so the *errors summary card* read
0 during the demo even though the recent-error badges and restart counts clearly flagged it — the
errored-state accent path is exercised whenever a proc actually reaches `errored`.)
