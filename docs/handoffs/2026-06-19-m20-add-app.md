## TL;DR

Added a dashboard flow to **create and launch a new app** on a connected agent — the true
`Start` op (distinct from M19's card "start" button, which only restarts an
already-stopped process). It is a thin layer over the existing start chain: a new
session-guarded `POST /api/apps` endpoint maps a request to `ControlOp_Start` and forwards
it via the existing `FleetController.Control`; the agent's existing
`handleFleetCommand`/`doStart`/`appSpecToConfig`/`manager.Add` path does the rest
(including persistence). The UI is an "+ add app" modal opened from the overview header.

A **`source` discriminator** in the request body (`{type:"command", ...}`) is the only
forward-compat concession to the **future full git-support milestone** the user flagged —
a `{type:"git", ...}` variant slots in additively later. No git logic was built.

## What changed this session

- **Backend** (`internal/dashboard/apps.go`, route in `handlers.go`): `POST /api/apps`
  decodes `{agent, source}`, switches on `source.type` (only `"command"` today → else
  400), builds `ControlOp_Start{StartRequest{[]AppSpec}}` via `startOp`, forwards through
  `Control`. Reuses `control.go`'s error mapping verbatim: 200 `{ok:true}`; 200
  `{ok:false,error}` for agent-rejected (e.g. duplicate name); 400 for unsupported
  source / missing `agent`/`name`/`cmd`; 401 no session; 502 agent unreachable. Audit log
  line `dashboard: add app <name> -> <agent> by <user>: ok=…`.
- **Frontend** (`web/src/api.ts`, `AddAppModal.tsx`, `Overview.tsx`, `styles.css`):
  `addApp(agent, source)` client (+ `CommandSource` type); a Signal-styled modal with a
  target-agent dropdown (connected-only, auto-select when one, hint when none), required
  name/cmd, common args/cwd/instances, and a collapsible **advanced** section (env
  key/value rows, restart select, max_restarts, kill_timeout); a `+ add app` button on the
  overview header. On success the modal closes and the new card appears on the next 2s
  fleet poll.
- **No** proto / agent / manager / supervisor changes. Per-app log-retention fields were
  intentionally left out of the form.

## Key decisions (non-obvious)

- **Dedicated `POST /api/apps`, not `action:"start"` on `/api/control`** — control is
  selector-based; app creation carries a full spec + a source. Separate endpoint keeps the
  concerns clean and is the natural seam for the future git source.
- **No web test framework added.** The repo has none (M15–M19 verified the UI via live
  demo). Frontend correctness = `tsc -b` (in `make ui`) + the live in-browser demo.
  Backend is full Go TDD.

## Live demo result (2026-06-19, scratch `/tmp/marshal-m20-demo`, server `:9000`/`:9001`)

Real fleet: server on `:9000`(gRPC)/`:9001`(dashboard) + a connected agent `dev-1` running
a `seed` app. Auth set while the server was down (password `demopass123`, enroll token
rotated, fingerprint pinned in the agent yaml).

- **Backend, real HTTP against the live fleet** (the exact new handler path):
  - add `demo2` (command source, `instances:2`, env, `restart:on-failure`) → `{ok:true}`;
    `/api/fleet` then showed **two** `demo2` instances `online`.
  - duplicate name → 200 `{ok:false,"app \"demo2\" already exists"}`.
  - `source.type:"git"` → 400 `unsupported source type`.
  - missing cmd → 400 `name and cmd required`.
  - unknown agent → 502.  no session → 401.
- **In-browser (Vite dev proxy `:5173` → `https://localhost:9001`, via the Preview tool):**
  logged in; overview rendered in Signal with the new `+ add app` button; opened the modal
  (target agent auto-selected `dev-1`, "add app" disabled until name+cmd set); filled
  name `added-via-ui`, cmd `/bin/sleep`, args `600`, submitted → modal closed and a new
  **`added-via-ui` online** card appeared (processes 4/4 up); re-opened, entered an existing
  name → red in-modal banner **`app "seed" already exists`**, modal stayed open.
- **Teardown:** agent + server + Vite stopped, scratch dir + `.claude/launch.json` removed,
  `web/vite.config.ts` reverted. `pgrep -fl marshal` shows only the user's own daemon
  (pid 1312, `clock`); no demo orphans; repo clean.

**Demo gotcha for next time:** the Preview/Vite proxy refused outbound to a non-standard
backend port (`:19001` → ECONNREFUSED while `curl` succeeded — likely the preview sandbox).
Running the demo server on the **conventional `:9000`/`:9001`** (what the committed
`vite.config.ts` proxy targets) worked immediately. Don't fight the proxy on odd ports.

## Deferred / known issues

- **Full git source** — only the `source.type` discriminator landed; the `{type:"git"}`
  variant (repo/branch/build → clone+build+run) is the next milestone the user flagged.
- **Minor (from final review, non-blocking):** numeric inputs (`instances`,
  `max_restarts`) use `Number()`, so non-numeric text → `NaN` → JSON `null` → backend
  default (benign); a `type="number"`/`isFinite` guard would make intent explicit.
- **Minor:** a 401 from `/api/apps` surfaces in the modal as generic `error 401` (matches
  `control()`; the 2s fleet poll owns real auth/logout).
- **Minor:** `onAdded` is a no-op — the new card appears via the 2s poll (≤2s lag) rather
  than an immediate refetch.
- Still open from earlier: per-app log-retention fields in the form; edit-an-app;
  one-click port open; light-mode; dashboard `/api/audit` view.
