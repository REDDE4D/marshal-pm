# M-B · Agent & host metadata — design spec

**Date:** 2026-06-23
**Program:** dashboard redesign (see `2026-06-23-dashboard-program-roadmap.md`). First data milestone.
**Branch:** `mB-agent-metadata` (off `dev`).
**Size:** small. **Baseline today:** PARTIAL (only agent name, connected, last-seen tracked).

## Goal

Persist and expose per-agent host metadata so the dashboard's agent band can show
`hostname · os/arch · marshal vX · IP · up Yd` instead of just the agent name. The agent already
sends its name + version in `Hello`; the server currently keeps only the name.

## Field set (confirmed)

hostname, IP/address, OS, arch, marshal version, host uptime. (CPU core count deferred to M-C.)

- **IP is server-derived** from the gRPC peer (`peer.FromContext`), not self-reported — trustworthy,
  no agent config, reflects the real connection source.
- **Host uptime** is carried as **`host_boot_unix`** (boot time, static), sent once in `Hello`; the
  dashboard computes `now − boot`. Avoids streaming a changing number.
- **marshal version** already flows in `Hello.marshal_version` — just stop dropping it.

## Changes

### 1. Proto (`proto/marshal/v1/fleet.proto`)
Extend two messages (next free field numbers):
```
message Hello {
  string agent_name      = 1;
  string marshal_version = 2;
  string hostname        = 3;  // os.Hostname()
  string os              = 4;  // runtime.GOOS
  string arch            = 5;  // runtime.GOARCH
  int64  host_boot_unix  = 6;  // host boot time (uptime = now - boot)
}

message AgentState {
  string agent_name      = 1;
  bool   connected       = 2;
  int64  last_seen_unix  = 3;
  repeated ProcInfo procs = 4;
  string hostname        = 5;
  string ip              = 6;  // server-derived from peer
  string os              = 7;
  string arch            = 8;
  string marshal_version = 9;
  int64  host_boot_unix  = 10;
}
```
**Regen note:** there is **no Makefile/buf proto target**; `internal/pb/*.pb.go` is generated by
hand with `protoc` (protoc-gen-go v1.36.11, protoc v7.35.0) against `proto/marshal/v1/`. The
implementer must run protoc and commit the regenerated `internal/pb`. **Add a documented gen step**
(a `make proto` target or `scripts/gen-proto.sh`) as part of this milestone so future proto changes
are reproducible — this is a known friction point.

### 2. Agent (`internal/fleet/client.go` ~159)
Populate the new `Hello` fields: `os.Hostname()` (best-effort; empty on error), `runtime.GOOS`,
`runtime.GOARCH`, host boot time. Boot time via `gopsutil/v3/host.BootTime()` (already a dependency)
— wrap so a failure leaves the field 0 (UI shows "—"). Version stays `c.version`.

### 3. Server (`internal/server/server.go` Hello handler ~73; `internal/server/registry.go`)
- Extract the agent's remote IP from `peer.FromContext(stream.Context())` (strip the port).
- Pass metadata + IP into the registry. Extend `agentEntry` with `hostname, ip, os, arch,
  marshalVersion string; hostBootUnix int64`. Either widen `Registry.Open` or add a
  `Registry.SetMeta(name, meta)` called right after `Open` — prefer a small `AgentMeta` struct to
  keep the signature clean.
- The registry's `AgentState`-building method (the `ListFleet`/snapshot path) copies the fields out.

### 4. Dashboard API + SPA
- The `/api/fleet` JSON view (dashboard `fleetView`) gains `hostname, ip, os, arch,
  marshal_version, host_boot_unix`.
- `web/src/api.ts` `Agent` type gains the fields.
- **Minimal** current-UI surfacing: extend the existing Overview agent header (`agent-head`) to show
  `hostname · os/arch · marshal vX · IP · up Yd` (uptime from `host_boot_unix`). Full visual
  treatment is M-A; keep this functional, not polished.

- **server_test:** Hello with metadata → registry populated; IP derived from a faked peer context;

- **dashboard fleet handler test:** the JSON view serializes the new fields.
- Agent-side: a small unit asserting Hello is built with hostname/os/arch/boot set (factor the
  builder so it's testable without a live connection).
